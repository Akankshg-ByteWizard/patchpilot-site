System Design

Architecture built for speed, safety, and repeatability.

PatchPilot combines shell integration, contextual retrieval, memory, and approval gates so terminal recovery feels fast without losing control.

Low-latency context capture Memory-backed recommendation engine Approval-first write pipeline

End-to-End Flow

From command failure to successful retry.

  1. Hook: shell trap catches non-zero exit and command details.
  2. Context: daemon reads error output and relevant file window.
  3. Memory: vector search returns top similar successful incidents.
  4. Reasoning: fix agent proposes shell command and or diff patch.
  5. Approval: user confirms, with strict auto-approval rules.
  6. Write-back: successful fix is logged and promoted into memory.

Core Modules

Each module is isolated so contributors can improve reliability without breaking the full pipeline.

Shell Hook Layer

Bash, zsh, and fish hooks emit structured payloads to a daemon socket.

Context Collector

Extracts language, error type, and nearest code region for grounding.

Prompt Builder

Injects relevant few-shot examples from prior successful fixes.

Approval Gate

Prevents silent destructive edits; every patch request is explicit.

Patcher and Runner

Applies backup-safe edits, runs commands, and retries failed workflows.

Session Logger

Tracks outcome, confidence, and metadata for audits and training loops.

Safety Principles

The default behavior is conservative by design: suggest clearly, execute deliberately, and always preserve rollback paths.

Approval First

File patches never run silently. Users must explicitly approve edits.

Backup Before Modify

Any changed file is copied to a backup before patching.

Policy Boundaries

Ignored sensitive paths and command allow-lists reduce accidental risk.

Transparent Feedback

Every recommendation includes confidence and reason, never opaque actions.

PatchPilot architecture: practical, transparent, and approval-first.
